## 1. 产品概念

### 1.1 产品定位

**核心定位**：数字化的共享财务总监（Shared CFO）

**使命**：为中小企业提供专业、准确、可追溯的税务政策咨询服务

### 1.2 核心价值

| 价值维度 | 说明 |
|----------|------|
| **准确性** | 100%基于政策原文，每句都有依据 |
| **可追溯性** | 每个结论都标注政策来源 |
| **推理透明** | 逐步展示分析逻辑 |
| **通俗易懂** | 原文+口语化解释 |
| **风险意识** | 明确提示不确定事项 |

### 1.3 产品形态

- 第一阶段：Web端查询工具（免费）
- 第二阶段：SaaS决策支持平台（付费）
- 最终目标：数字化共享财务总监平台

### 2.2 两阶段客户策略

#### 阶段一：中小企业财务人员

**用户画像**：
- 年龄：25-40岁
- 专业：有财务基础，但税务专业度有限
- 痛点：政策更新快、查询耗时、担心理解错误
- 期望：快速、准确、有依据

**核心需求**：
| 需求类型 | 具体内容 |
|----------|----------|
| 智能查询 | 自然语言查询政策 |
| 政策原文 | 完整政策原文展示 |
| 操作指引 | 申报表填写、材料准备 |
| 收藏管理 | 个人知识库 |

#### 阶段二：中小企业老板/高管

**用户画像**：
- 年龄：30-50岁
- 专业：非财务背景，关注经营决策
- 痛点：看不懂财务报表、担心税务风险
- 期望：简单、易懂、有风险提示

**核心需求**：
| 需求类型 | 具体内容 |
|----------|----------|
| 场景化引导 | 用业务场景引导提问 |
| 健康诊断 | 税务风险体检 |
| 量化影响 | 用金额而非百分比说话 |
| 通俗解释 | 非专业人士版本 |

### 2.3 目标客户优先级

| 优先级 | 客户群体 | 理由 |
|--------|----------|------|
| P0 | 中小企业财务人员 | 高频使用、数据积累基础 |
| P1 | 中小企业老板/高管 | 付费意愿强、决策价值高 |
| P2 | 代账公司/税务师事务所 | 可以作为合作伙伴 |
| P3 | 大企业财务团队 | 定制化需求、高客单价 |
| P4 | 个人纳税人 | 低频、低付费意愿 |

## 3. 核心业务规则

### 3.1 答案生成规则

**固定输出格式**：

```
【结论】
XXX

【政策依据】
1. 《文件名称》（文号）第X条：
   "原文引用..."
2. 《文件名称》（文号）第X条：
   "原文引用..."

【分析过程】
基于上述政策，推理如下：
第一步：...
第二步：...
...

【通俗解释】
简单来说，就是...

【有效期】
政策执行期限：XXXX年XX月XX日至XXXX年XX月XX日
（如为现行有效政策，标注"现行有效"）

【风险提示】（如有）
本问题涉及XXX方面，实际执行中可能存在地区差异，建议咨询当地税务机关确认。
```

### 3.2 核心原则

| 原则 | 说明 |
|------|------|
| **严谨性** | 只能基于提供的政策原文回答，严禁添加个人解读或推测 |
| **推理性** | 分析过程要一步步展示逻辑链条 |
| **通俗性** | 引用原文后要用口语化方式解释 |
| **完整性** | 必须按固定格式输出 |
| **诚实性** | 检索不到相关政策时，明确告知"未找到相关政策" |
| **时效性** | 仅使用有效期内的政策 |
| **风险意识** | 对边界模糊、执行口径不一致的问题，给出风险提示 |

### 3.3 产品边界

**我们做什么**：
- ✓ 帮您快速找到相关政策
- ✓ 帮您理解政策条款的含义
- ✓ 帮您分析不同选择的税务影响
- ✓ 帮您识别需要注意的风险点

**我们不做什么**：
- ✗ 不替代专业的税务师
- ✗ 不保证建议的绝对正确（政策可能变化）
- ✗ 不承担因使用建议产生的法律责任
- ✗ 不提供申报操作等代办服务

**建议的使用方式**：
1. 用AI快速了解问题
2. 用AI形成初步判断
3. 带着AI的分析结果去咨询税务师
4. 让税务师做最终确认
5. 税务师负责申报和承担责任

## 4. 技术架构方案

### 4.1 技术栈选型

| 组件 | 选择 | 理由 |
|------|------|------|
| **后端语言** | Python | 爬虫生态（Scrapy、Playwright）、AI生态成熟 |
| **Web框架** | FastAPI | 异步高性能，原生支持Swagger文档 |
| **爬虫框架** | Scrapy + Playwright | Scrapy适合批量结构化抓取，Playwright处理动态JS |
| **向量数据库** | Qdrant | 轻量、开源友好、性能好，有云端版本 |
| **文档数据库** | MongoDB | 存储原始政策文档，Schema灵活 |
| **前端框架** | Vue3 + Element Plus | 国内税务用户习惯，组件成熟 |
| **大模型** | GLM-4-Flash / GLM-4-Air | 便宜好用，Flash适合高频调用 |
| **RAG框架** | LangChain 或 LlamaIndex | 生态成熟，GLM兼容性好 |
| **部署** | 阿里云/腾讯云 | 国内网络环境好，GLM调用延迟低 |

### 4.3 RAG系统核心流程

```
用户提问
    ↓
提取关键词
    ↓
向量检索（从Qdrant获取相关政策和问答）
    ↓
拼接Prompt（检索结果 + 用户问题 + 业务规则）
    ↓
GLM推理
    ↓
返回结果（标注引用来源）
```

### 5.2 阶段化部署策略

1. **第一阶段（1-2个月）**：用入门版
   - 目标：验证方案可行性，跑通全流程
   - 重点：爬虫数据质量、RAG效果调优
   - 投入：~¥800

2. **第二阶段（3-6个月）**：升级标准版
   - 目标：小范围试运营，收集真实用户反馈
   - 重点：优化Prompt、提升答案质量
   - 投入：~¥8000

3. **第三阶段（6个月后）**：根据用户量决定
   - 用户量稳定 → 继续标准版
   - 快速增长 → 考虑专业版

### 5.3 成本优化建议

- Qdrant可以用开源版自建，节省云端费用
- GLM-4-Flash已经足够好用，不必强求Plus/Air
- 非高峰期可以降低服务器配置

4. 税务师基于AI分析提供专业意见
5. 后续服务由税务师承接

**收益模式**：
- AI：技术服务费（月租）
- 税务师：咨询服务费（按次）
- 平台：撮合抽成（10-20%）

## 9. 关键问题与解决方案

### 9.1 责任归属问题

#### 问题描述

税务决策涉及法律责任，AI无法承担法律责任。老板为什么最终还是找税务师？

#### 深度原因分析

| 原因 | 说明 | AI情况 | 税务师情况 |
|------|------|---------|-----------|
| 责任主体 | 谁来承担后果 | ❌ 无 | ✅ 执业保险+监管 |
| 专业资质 | 专业背书 | ❌ 无 | ✅ 执业资格 |
| 经验积累 | 可追溯经验 | ❌ 未知 | ✅ 案例可查 |
| 人际互动 | 长期关系 | ❌ 无 | ✅ 长期合作 |
| 风险共担 | 一起承担风险 | ❌ 无 | ✅ 执业保险 |

#### 解决方案

**方案1：AI+专业机构混合模式**

```
┌──────────────┐      ┌──────────────┐
│    AI系统    │ ───→ │ 合作税务师   │
│              │      │   事务所      │
└──────────────┘      └──────────────┘
     │                      │
     ▼                      ▼
• 快速响应              • 专业审核
• 24/7服务              • 责任承担
• 低成本查询            • 执业保险
• 初步筛选              • 申报操作
```

**方案2：责任保险机制**

- 与保险公司合作推出"AI税务咨询责任险"
- 保障范围：因AI建议导致的税务罚款、补税+滞纳金
- 免责条款：用户信息不准确、未完全执行、政策变化

**方案3：分阶段产品策略**

- 信息查询工具 → 决策支持工具 → 专业服务平台
- 明确产品边界，管理用户预期

**方案4：明确产品边界 + 用户教育**

- 明确定位为"辅助工具"而非"专业咨询"
- 建议的正确使用方式：
  1. 用AI快速了解问题
  2. 用AI形成初步判断
  3. 带着AI的分析结果去咨询税务师
  4. 让税务师做最终确认

### 9.2 关键技术挑战

| 挑战 | 风险等级 | 解决方案 |
|------|----------|----------|
| 反爬虫机制 | 中 | 控制频率、使用代理、设置User-Agent |
| 网站结构变化 | 中 | 建立监控机制 |
| 数据去重 | 低 | 基于URL+标题+内容哈希去重 |
| PDF附件解析 | 中 | 使用pdfplumber/PyPDF2工具 |
| 向量化成本 | 低 | 使用GLM-4-Flash的embedding API |
| 政策时效性 | 高 | 建立有效期字段，过期政策不参与检索 |
| 地区差异 | 中 | 答案中标注"各地执行口径可能不同" |
| 政策理解偏差 | 中 | Prompt强调"严格按原文回答" |

### 9.3 合规风险

| 风险 | 风险等级 | 解决方案 |
|------|----------|----------|
| 数据版权 | 低 | 政府数据可公开使用，注明来源 |
| 答案责任 | 高 | 明确免责声明，标注"仅供参考" |
| 隐私保护 | 中 | 用户数据脱敏处理 |
